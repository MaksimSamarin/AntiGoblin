# Troubleshooting

Короткая памятка по уже найденным проблемам и решениям.

## 1. Игра не заходит на сервер, висит логин или `HANDSHAKE_DEADLINK`

Симптомы:

- игра долго подключается к серверу;
- зависает на экране подключения;
- может показывать `HANDSHAKE_DEADLINK`.

Причина:

- игровой `UDP` ломается, если пускать его через `xray` по `TPROXY`.

Решение:

- `xkeen_udp` должен быть выключен;
- `UDP` должен идти напрямую, мимо `xray`.

Проверка:

- в `iptables -t mangle -S PREROUTING` не должно быть наших `xkeen_udp`-хуков;
- в runtime не должно быть цепочки `xkeen_udp`.

## 2. `chatgpt.com` не открывается

Симптомы:

- сайт не открывается;
- может показывать `Please try again later` или Cloudflare-страницу.

Что уже выяснили:

- проблема может быть не в домене как таковом, а в текущем routing/выходе;
- если `chatgpt.com/cdn-cgi/trace` показывает VPN IP, значит домен реально идет через VPN;
- если в браузере при этом виден RU IP, значит запрос уходит другим путем.

Практика:

- сначала проверять живой `05_routing.json`;
- потом проверять `cdn-cgi/trace`;
- отдельно учитывать локальные VPN-клиенты на ПК (`v2rayN`, `sing-box`, `TUN`).

## 3. Mi Home / аэрогриль очень долго подключается

Симптомы:

- устройство управляется 3-5 минут;
- при этом локалка вроде бы работает.

Что выяснили:

- проблема не только в локальном discovery;
- проблема может оставаться даже если `UDP` идет напрямую;
- локальный bypass для `192.168.2.0/24` и multicast нужен, но он не гарантирует решение;
- для таких приложений сам факт `TCP REDIRECT -> xray -> direct/vless` может быть уже проблемой.

Текущий вывод:

- полное устройство в `xkeen` не всегда подходит для IoT/облачных приложений;
- если Mi Home критичен, телефон лучше не держать в полном `xkeen`, либо отдельно проектировать selective-модель.

## 4. После изменения групп/политик в Keenetic все ломается

Симптомы:

- VPN внезапно перестает работать после изменения политики;
- UI может открываться, но трафик не идет.

Причина:

- Keenetic может пересобрать `iptables`, и наши runtime-hooks пропадают.

Решение:

- `xkeen-selfheal.sh` должен быть жив;
- cron должен запускать self-heal несколько раз в минуту;
- рабочий runtime сейчас:
  - `PREROUTING mark 0xffffaaa -> xkeen`
  - `xkeen`: локальные `RETURN` + `TCP REDIRECT 61219`

## 5. Что сейчас считается правильной схемой

Текущая рабочая схема:

- `HydraRoute` удален полностью;
- группа Keenetic `xkeen` использует mark `0xffffaaa`;
- весь `TCP` устройств из `xkeen` идет в `xray`;
- `UDP` идет напрямую;
- локалка и discovery идут через `RETURN`;
- `xkeen_udp` выключен;
- `xkeen_quic` выключен.

## 6. Что не делать снова

- не включать обратно `xkeen_udp` без явной причины;
- не включать обратно `xkeen_quic` широким правилом;
- не возвращать `HydraRoute`;
- не править живые JSON на роутере через кривую сериализацию PowerShell-объектов.

## 7. Временный инструмент отладки: `xray access log`

Когда нужен:

- если надо понять, какие реальные `IP:port` телефона или ПК еще заходят в `xray`;
- если счетчиков `iptables` уже недостаточно;
- если нужно отличить `RETURN` до `xray` от `redirect -> direct/vless` внутри `xray`.

Как включать:

- временно поменять `/opt/etc/xray/configs/01_log.json`;
- выставить:
  - `"loglevel": "info"`
- очистить:
  - `/opt/var/log/xray/access.log`
- перезапустить `xray`.

Что смотреть:

- строки вида:
  - `from 192.168.2.91:... accepted tcp:IP:PORT [redirect -> ...]`
- это показывает:
  - источник;
  - целевой `IP:port`;
  - куда `xray` отправил трафик:
    - `direct`
    - `vless-reality`

Как использовать правильно:

- включать только на короткое окно захвата;
- перед тестом очищать `access.log` в ноль;
- во время захвата выполнять только один сценарий:
  - например, только Mi Home / аэрогриль;
- потом сразу снимать лог и разбирать только свежий хвост.

Что это не показывает:

- трафик, который уже ушел в `RETURN` до `xray`;
- для такого трафика нужны счетчики `iptables`/`ipset`.

Как выключать обратно:

- вернуть в `/opt/etc/xray/configs/01_log.json`:
  - `"loglevel": "none"`
- перезапустить `xray`.

Важно:

- не оставлять `access log` включенным надолго;
- после отладки всегда возвращать `loglevel: none`.
