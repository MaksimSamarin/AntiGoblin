# Troubleshooting

Сюда заносятся только подтвержденные проблемы, их реальные причины и рабочие решения.

Для каждой решенной проблемы фиксируй:

- симптом;
- корневую причину;
- как проверяли;
- рабочее решение;
- что не надо повторять.

## 1. Игра не заходит на сервер или висит на логине

### Симптом

- долгий логин;
- зависание на подключении к серверу;
- `HANDSHAKE_DEADLINK`.

### Причина

Игровой `UDP` ломается, если пускать его через `xray` по `TPROXY`.

### Рабочее решение

- держать `xkeen_udp` выключенным;
- пускать `UDP` напрямую.

### Проверка

- в `iptables -t mangle -S PREROUTING` не должно быть активных `xkeen_udp`-хуков;
- в runtime не должно быть цепочки `xkeen_udp`.

### Что не надо повторять

Не включать `xkeen_udp` обратно как общий способ “починить сеть”.

## 2. `chatgpt.com` не открывается

### Симптом

- браузер показывает ошибку;
- может появляться Cloudflare-страница с direct IP или challenge.

### Причина

Проблема может быть не в одном месте, а в нескольких слоях:

- router routing path;
- состояние браузера или старый challenge;
- локальный VPN-клиент на ПК или `TUN`;
- репутация выходного VPN IP.

### Проверка

Проверять по порядку:

1. live `05_routing.json`;
2. `https://chatgpt.com/cdn-cgi/trace`;
3. состояние браузера и локального VPN-клиента на ПК.

### Наблюдение

Если `cdn-cgi/trace` показывает VPN IP, значит router path активен. Если браузер при этом показывает direct IP, значит есть еще один путь поверх маршрутизации роутера.

## 3. Mi Home или Xiaomi-аэрогриль подключаются по 3-5 минут

### Симптом

- управление устройством занимает минуты;
- локалка при этом вроде жива.

### Причина

Проблема не только в локальном discovery.

Даже если `UDP` идет напрямую и локалка вынесена в bypass, некоторые мобильные и IoT-сценарии ломаются уже от самого факта прозрачного `TCP REDIRECT -> xray`, даже если внутри `xray` поток потом идет в `direct`.

### Что подтверждено

- локальный bypass обязателен;
- multicast и broadcast нужно bypass-ить;
- одного `UDP direct` недостаточно;
- для части Xiaomi cloud-трафика `RETURN` до `xray` правильнее, чем `direct` внутри `xray`.

### Практика

- не считать `direct` и `RETURN` одним и тем же;
- для таких кейсов включать временный `xray access log`;
- подтвержденные cloud-направления выносить в `Bypass`.

## 4. После изменения политик Keenetic пропадает VPN-path

### Симптом

- после переноса устройств между политиками Keenetic VPN path исчезает;
- UI может открываться, но трафик уже не идет как надо.

### Причина

Keenetic может пересобирать `iptables`, и тогда наши runtime-hooks пропадают.

### Рабочее решение

- держать `xkeen-selfheal.sh` активным;
- держать watchdog `S25antigoblin-selfheal` активным;
- cron можно оставлять как дополнительный слой, но не как основной механизм восстановления;
- проверять, что жив hook:
- `PREROUTING mark <mark_xkeen> -> xkeen`.

## 5. После `Сохранить и применить` сервисы ведут себя по-старому

### Симптом

- в UI группы уже переключены;
- в `xkeen-ui-state.json` и `05_routing.json` изменения уже записаны;
- но фактическое поведение сервисов остается старым;
- пример: `2ip` уже переведен в `direct`, а по факту все еще показывает VPN IP.

### Причина

Проблема была не в самих группах, а в restart `xray`.

`05_routing.json` обновлялся, но процесс `xray` мог остаться со старой конфигурацией в памяти. Из-за этого:

- файлы на диске уже новые;
- runtime-поведение еще старое.

Дополнительно мешали stale pid-файлы:

- `/opt/var/run/xray-ui.pid`;
- `/opt/var/run/xray.pid`.

### Как проверить

Смотреть по слоям:

1. `xkeen-ui-state.json`;
2. `05_routing.json`;
3. живой процесс `xray`;
4. фактическое поведение сервиса.

Если первые два слоя уже новые, а поведение старое, значит надо подозревать stale runtime `xray`.

### Рабочее решение

- делать реальный restart `xray` сразу после успешной валидации конфигов;
- перед restart удалять stale pid-файлы;
- после restart убеждаться, что `61219` снова слушается.

### Что не надо повторять

- нельзя считать apply завершенным только потому, что `05_routing.json` уже переписан;
- нельзя оставлять restart `xray` на отложенную и ненадежную схему.

## 6. Временный инструмент отладки: `xray access log`

Использовать только короткими сессиями.

### Когда включать

- когда счетчиков `iptables` уже недостаточно;
- когда нужен реальный `IP:port`, который входит в `xray`;
- когда надо увидеть, ушел поток в `vless-reality` или в `direct`.

### Как включить

В `/opt/etc/xray/configs/01_log.json` выставить:

```json
{
  "log": {
    "loglevel": "info"
  }
}
```

Потом:

- очистить `/opt/var/log/xray/access.log`;
- перезапустить `xray`.

### Что показывает

Строки вида:

```text
from 192.168.2.91:... accepted tcp:IP:PORT [redirect -> ...]
```

Это дает:

- источник;
- destination `IP:port`;
- итоговый outbound внутри `xray`.

### Чего не показывает

Не показывает трафик, который уже ушел через `RETURN` до `xray`.

Для этого нужны счетчики `iptables` и `ipset`.

### Как выключить обратно

Вернуть:

```json
{
  "log": {
    "loglevel": "none"
  }
}
```

Потом перезапустить `xray`.

### Правило

Нельзя оставлять access log включенным дольше, чем нужно для диагностики.

## 7. Проверка логов после аварий и отладки

### Симптом

- после серии отладок или аварийного восстановления есть риск оставить включенные логи;
- это может постепенно засорять флешку `/opt`.

### Что должно быть в норме

`/opt/etc/xray/configs/01_log.json` должен содержать только:

```json
{
  "log": {
    "loglevel": "none"
  }
}
```

Активные логи `xray` должны быть пустыми или почти пустыми.

### Что проверять

- `/opt/etc/xray/configs/01_log.json`;
- `/opt/var/log/xray-access.log`;
- `/opt/var/log/xray-error.log`;
- `/opt/var/log/xkeen-selfheal.log`.
- `/opt/var/log/xkeen-health.log`.

### Рабочее решение

- вернуть `loglevel: "none"`;
- при необходимости обнулить старые debug-хвосты;
- проверить конфиг через `xray run -test -confdir /opt/etc/xray/configs`;
- перезапустить `xray`.

## 8. Через 1-2 дня все “встает”, а reboot помогает

Симптом:

- часть сайтов или сервисов перестает работать;
- `self-heal` формально жив;
- reboot роутера сразу возвращает рабочее состояние.

Причина:

- старой проверки “порт `61219` слушается” недостаточно;
- `xray` может деградировать, но не падать полностью;
- возможны:
  - `probe_fail` при открытом порту;
  - критический рост `fd` у `xray`;
  - высокий `conntrack`;
  - низкий запас памяти.

Рабочее решение:

- `self-heal` должен вести отдельный health-log:
  - `/opt/var/log/xkeen-health.log`
- в нормальном режиме он должен писать health-snapshot примерно раз в 5 минут
- `self-heal` должен проверять:
  - наличие `xray` pid
  - факт прослушивания `61219`
  - количество `fd` у процесса `xray`
  - `nf_conntrack_count / nf_conntrack_max`
  - `MemAvailable / MemTotal`
- если `fd > 600` держится 3 цикла подряд, допускается controlled restart `xray` с cooldown.
- если у `xray` копится слишком много `FIN_WAIT1/FIN_WAIT2` к VPN-серверу, это тоже считается деградацией runtime.
- для этого в health-log дополнительно пишутся метрики outbound-сокетов к `vless-reality` серверу:
  - сколько `ESTABLISHED`
  - сколько `FIN_WAIT1/FIN_WAIT2`
  - сколько всего сокетов на VPN-апстрим
- отдельно пишется `vpn_orphan_fin`: сколько `FIN_WAIT1/FIN_WAIT2` к текущему VPN host/port уже не принадлежит живому процессу `xray`.
- `vpn_orphan_fin` используется как диагностическая метрика, а не как самостоятельный restart-триггер.
- После controlled restart self-heal пытается точечно удалить conntrack-записи к текущему VPN host/port, если установлен Entware-пакет `conntrack`.
- Важно: orphan `FIN_WAIT1/FIN_WAIT2` часто являются следствием restart `xray` или массового переключения устройств между группами. Их нельзя использовать как единственную причину нового restart, иначе self-heal может сам создать restart-loop.

Что проверять:

- `/opt/var/log/xkeen-health.log`
- `/opt/var/log/xkeen-selfheal.log`
- `/proc/sys/net/netfilter/nf_conntrack_count`
- `/proc/sys/net/netfilter/nf_conntrack_max`

Что не надо повторять:

- нельзя считать `netstat :61219` достаточной health-check проверкой `xray`.

## 9. После переноса устройства между группами сайты продолжают идти старым путем

### Симптом

- устройство перенесли из `xkeen` в другую группу или обратно;
- часть сайтов продолжает вести себя так, будто устройство осталось в старой группе;
- после перезапуска роутера или `xray` поведение нормализуется.

### Причина

Keenetic меняет policy/mark для новых соединений, но старые TCP/NAT-записи остаются в conntrack до истечения таймаута. Из-за этого браузер может держать старые сессии по старому пути, особенно для HTTP/2, YouTube, ChatGPT, Claude и других сервисов с большим количеством долгих TCP-соединений.

### Решение

- self-heal отслеживает `vpn_orphan_fin`;
- `vpn_orphan_fin` показывает хвосты старых TCP-сессий после переключения policy или restart `xray`;
- restart по одному только `vpn_orphan_fin` запрещен, потому что restart сам может создавать такие orphan-сокеты;
- после controlled restart по реальной причине (`fd_critical`, `vpn_fin_critical`, падение xray) self-heal удаляет conntrack-записи к текущему VPN endpoint через пакет `conntrack`;
- bootstrap и backend deploy устанавливают Entware-пакет `conntrack`.

## 10. После переноса устройств между политиками Keenetic все ломается

### Симптом

- после переноса устройств между политиками в Keenetic пропадает интернет или VPN-path;
- в `iptables` исчезает `PREROUTING -> xkeen`;
- `xkeen-selfheal.sh --force` может возвращать `0`, но ничего не чинить.

### Причина

Keenetic пересобирает `iptables`, а наш self-heal может не сработать из-за stale lock:

- `/tmp/xkeen-selfheal.lock`

В этом случае новый запуск скрипта сразу выходит, как будто все успешно, хотя runtime уже разрушен.

### Как проверить

Проверять:

1. есть ли `PREROUTING -m connmark --mark <mark_xkeen> ... -j xkeen`;
2. жив ли `xray` и слушается ли `61219`;
3. не остался ли stale lock в `/tmp/xkeen-selfheal.lock`.

### Рабочее решение

- lock должен хранить `pid`;
- если процесс с этим `pid` уже не жив, lock нужно считать stale;
- stale lock нужно удалять автоматически и только потом запускать repair.
- apply из UI и self-heal должны использовать одну общую реализацию `/opt/share/xkeen-manager/api/xkeen-runtime.sh`;
- runtime repair должен цепляться только к mark текущей политики `description xkeen`, без hardcoded mark старых роутеров.

## 10.1. Через день-два CDN-домены начинают идти не туда

### Симптом

- после долгой работы часть сайтов внезапно перестает идти ожидаемым путем;
- после ручного apply, restart или reboot поведение возвращается;
- проблема чаще заметна на Cloudflare/AWS/Discord/YouTube, где IP могут меняться.

### Причина

`xkeen_bypass` и `xkeen_udp_route` строятся из доменов через DNS. Если IP у CDN поменялись, старый `ipset` может стать устаревшим, хотя `xray`, `iptables` и политика `xkeen` формально живы.

### Решение

- self-heal периодически пересобирает runtime-наборы примерно раз в 5 минут;
- пересборка не должна рестартовать `xray`, если сам `xray` здоров;
- пересборка должна использовать `ipset swap`, чтобы не оставлять окно с пустым bypass-набором.

## 11. После чистой установки `xkeen` есть в CLI, но не видна в Keenetic UI

Симптом:

- `show running-config` и `show ip policy` показывают политику `xkeen`;
- но на странице `Приоритеты подключений` в Keenetic UI она не видна.

Причина:

- для UI-видимой дополнительной политики оказалось недостаточно просто создать `Policy1`, `Policy2` и так далее;
- рабочий install-path для Keenetic UI — создавать `xkeen` как дополнительную политику вида `Policy42+`.

Решение:

- bootstrap должен создавать `xkeen` как `Policy42+`;
- у политики должен быть `description xkeen`;
- у политики должен быть `permit global <WAN>`;
- после создания обязательно нужен `system configuration save`;
- mark нужно определять динамически по `description xkeen`, а не хардкодить.

Важно:

- это особенно важно, если у пользователя уже есть свои политики, например `no_vpn`;
- `AntiGoblin` не должен цепляться к чужим policy mark.
- привязка устройства через `ip hotspot host ... policy Policy42` для появления `xkeen` в UI не обязательна.

## 12. Пользователь удалил `xkeen` в Keenetic UI

Симптом:

- политика `xkeen` пропала из Keenetic UI;
- устройства перестали попадать в AntiGoblin runtime;
- `show ip policy` больше не показывает `description = xkeen`.

Решение:

- `xkeen-selfheal.sh` должен создать `xkeen` заново как `Policy42+`;
- после создания он должен сделать `system configuration save`;
- затем он должен заново определить mark новой `xkeen`;
- после этого он должен восстановить `PREROUTING -> xkeen` только на mark этой политики.

Проверка:

1. в `show ip policy` снова есть `description = xkeen`;
2. в saved config есть `ip policy Policy42` с `description xkeen`;
3. в `iptables-save -t nat` есть hook только на mark `xkeen`;
4. существующие пользовательские политики, например `no_vpn`, остались без изменений.

Наблюдение:

- надежный автоподъем после удаления `xkeen` обеспечивает watchdog `S25antigoblin-selfheal`, который гоняет `xkeen-selfheal.sh` каждые 15 секунд;
- на cron как на единственный источник авто-восстановления полагаться нельзя.

### Что не надо повторять

- нельзя считать `EXIT:0` от self-heal доказательством, что repair реально отработал;
- нельзя держать lock без проверки, жив ли владеющий процесс.

## 13. Chrome, Claude или Discord обходят VPN через QUIC

### Симптом

- TCP-запросы домена попадают в `xray`, но сайт в браузере все равно идет напрямую;
- `google-chrome --disable-quic` сразу чинит проблему;
- в `conntrack` или `tcpdump` видны `UDP/443` соединения от устройства из политики `xkeen`.

### Причина

Архитектура AntiGoblin перехватывает только `TCP`. Это сделано специально: общий `UDP -> xray` через старую TPROXY-схему ломал игры, IoT и часть мобильных сценариев. Но современные клиенты могут использовать QUIC поверх `UDP/443`; такой трафик не проходит через TCP `REDIRECT -> 61219` и может обходить VPN.

### Решение

Отдельная профильная галка `Блокировать UDP/443 (QUIC)` удалена. Вместо нее используется точечный флаг `UDP через VPN` у конкретной UI-группы: он проксирует только UDP к IP из доменов/CIDR этой группы.

Важно: это не возвращает старый общий `xkeen_udp` и не включает UDP-проксирование для всей политики `xkeen`. Старые цепочки `xkeen_udp` и `xkeen_quic` остаются retired-хвостами и должны вычищаться self-heal.

### Проверка

```sh
ipset list xkeen_udp_route
iptables -t mangle -S PREROUTING | grep xkeen_udp_route
iptables -t mangle -S PREROUTING | grep -E 'xkeen_udp|xkeen_quic'
```

В `mangle/PREROUTING` не должно быть активных `xkeen_udp` или `xkeen_quic` хуков.

## 14. Discord voice/video требует UDP через VPN

### Симптом

- текстовый Discord и сайт открываются через VPN;
- голосовая связь подключена, но ping огромный или медиа не работает;
- в `conntrack` видны UDP-сессии к Cloudflare/Discord RTC, например `104.29.x.x:19300-19400`.

### Причина

Discord voice/video использует UDP RTC. Если UDP оставлен direct, этот трафик может не пройти у провайдера или в текущем регионе, даже когда TCP-часть Discord уже идет через VPN.

### Решение

Включить у Discord-группы в UI флаг `UDP через VPN`. Это не возвращает старый общий `xkeen_udp`: self-heal собирает `xkeen_udp_route` только из включенных групп и TPROXY-проксирует только UDP к IP из этого набора.

Для Discord voice лучше добавлять CIDR/IP диапазоны, а не только домены. У RTC UDP-пакетов часто нет домена, который `xray` сможет сопоставить routing-правилу; IP/CIDR-правило работает предсказуемее.

Важно: `05_routing.json` должен содержать отдельное правило `inboundTag: ["tproxy"]`, `network: "udp"`, `outboundTag: "vless-reality"`. Иначе UDP попадет в `xray`, но может уйти по fallback/direct, потому что у пакета нет домена для обычного domain-rule.

Для VLESS Reality поверх TCP outbound должен использовать XUDP:

```json
"mux": {
  "enabled": true,
  "concurrency": -1,
  "xudpConcurrency": 16,
  "xudpProxyUDP443": "skip"
}
```

`concurrency: -1` оставляет обычный TCP без Mux, а `xudpConcurrency` включает упаковку UDP. Без этого Discord UDP может попадать в `xray`, но голос все равно будет висеть с огромной задержкой.

### Проверка

```sh
ipset list xkeen_udp_route
iptables -t mangle -L xkeen_udp_route -v -n
iptables -t mangle -S PREROUTING | grep xkeen_udp_route
ip rule show | grep 'fwmark 0x111'
ip route show table 111
netstat -lnptu | grep 61220
```

Если счетчик `xkeen_udp_route` растет, UDP действительно попадает в `xray` через `61220`.
