# Troubleshooting

Сюда заносятся подтвержденные баги, реальные причины и рабочие решения.

Для каждой решенной проблемы нужно фиксировать:

- симптом
- корневую причину
- способ проверки
- рабочее решение
- что не надо повторять

## 1. Игра не заходит на сервер или висит на логине

### Симптом

- долгий логин
- зависание на подключении к серверу
- `HANDSHAKE_DEADLINK`

### Причина

Игровой `UDP` ломается, если пускать его через `xray` по `TPROXY`.

### Рабочее решение

- держать `xkeen_udp` выключенным
- пускать `UDP` напрямую

### Проверка

- в `iptables -t mangle -S PREROUTING` не должно быть активных `xkeen_udp`-хуков
- в runtime не должно быть цепочки `xkeen_udp`

### Что не повторять

Не включать `xkeen_udp` обратно как общий способ “починить сеть”.

## 2. `chatgpt.com` не открывается

### Симптом

- браузер показывает ошибку
- может появляться Cloudflare-страница с direct IP или challenge

### Причина

Проблема может быть не в одном месте, а в нескольких слоях:

- неверный router routing path
- состояние браузера или старый challenge
- локальный VPN-клиент на ПК или `TUN`
- репутация выходного VPN IP

### Проверка

Проверять по порядку:

1. live `05_routing.json`
2. `https://chatgpt.com/cdn-cgi/trace`
3. состояние браузера и локального VPN-клиента на ПК

### Наблюдение

Если `cdn-cgi/trace` показывает VPN IP, значит router path активен. Если браузер при этом показывает direct IP, значит есть еще один путь поверх маршрутизации роутера.

## 3. Mi Home или аэрогриль Xiaomi подключаются по 3-5 минут

### Симптом

- управление устройством занимает 3-5 минут
- локалка при этом вроде жива

### Причина

Проблема не только в локальном discovery.

Даже если `UDP` идет напрямую и локалка вынесена в bypass, некоторые мобильные или IoT-сценарии ломаются уже от самого факта прозрачного `TCP REDIRECT -> xray`, даже если внутри `xray` поток потом уходит в `direct`.

### Что уже подтверждено

- локальный bypass обязателен
- multicast и broadcast нужно bypass-ить
- одного `UDP direct` недостаточно
- для части Xiaomi cloud-трафика `RETURN` до `xray` правильнее, чем `direct` внутри `xray`

### Практика

- не считать `direct` и `RETURN` одним и тем же
- для таких кейсов включать временный `xray access log`
- подтвержденные cloud-направления выносить в runtime bypass, если их надо полностью убрать из `xray`

## 4. VPN пропадает после изменения политик или групп в Keenetic

### Симптом

- VPN path исчезает после переноса устройств между политиками Keenetic
- UI может открываться, но трафик уже не идет

### Причина

Keenetic может пересобрать `iptables`, и тогда наши runtime-hooks пропадают.

### Рабочее решение

- держать `xkeen-selfheal.sh` активным
- держать cron entries активными
- следить, чтобы live-hook был таким:
  - `PREROUTING mark 0xffffaaa -> xkeen`

## 5. Временный инструмент отладки: `xray access log`

Использовать только короткими сессиями.

### Когда включать

- когда счетчиков `iptables` уже недостаточно
- когда нужен реальный `IP:port`, который входит в `xray`
- когда надо увидеть, ушел поток в `vless-reality` или в `direct`

### Как включить

В `/opt/etc/xray/configs/01_log.json` выставить:

```json
{
  "log": {
    "loglevel": "info"
  }
}
```

После этого:

- очистить `/opt/var/log/xray/access.log`
- перезапустить `xray`

### Что показывает

Строки вида:

```text
from 192.168.2.91:... accepted tcp:IP:PORT [redirect -> ...]
```

Это дает:

- источник
- destination `IP:port`
- итоговый outbound внутри `xray`

### Чего не показывает

- трафик, который уже ушел через `RETURN` до `xray`

Для этого трафика нужны счетчики `iptables` и `ipset`.

### Как выключить обратно

Вернуть:

```json
{
  "log": {
    "loglevel": "none"
  }
}
```

Потом перезапустить `xray`.

### Правило

Нельзя оставлять access log включенным дольше, чем нужно для диагностики.
